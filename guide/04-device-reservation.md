# 占用设备

设备列表是查询时刻的结果。多个座席可能同时看到同一台设备“可用”，如果都继续发起呼叫，就会出现并发冲突。生产集成中，接入方后端必须在“选择设备”和“发布 CALL”之间实现设备占用。

## 为什么不能只靠设备状态

设备端可以在忙线时返回 `DEVICE_BUSY`，但这已经是比较晚的失败路径。更好的做法是在业务后端先做互斥：

1. 座席选择设备并点击呼叫。
2. 后端尝试占用设备。
3. 占用成功才继续获取 Token、连接 MQTT、发布 CALL。
4. 占用失败则提示设备已被使用，要求座席刷新或重选。

这样可以减少无效 MQTT/RTC 准备，也能让通话记录更清晰。

## 推荐占用状态

| 状态 | 说明 |
| --- | --- |
| `reserved` | 座席已选中设备，后端已占用，正在准备呼叫 |
| `calling` | CALL 已发布，等待设备状态 |
| `in_call` | 已收到 `ANSWERED`，通话中 |
| `released` | 呼叫结束或失败，设备已释放 |

建议占用记录至少保存：

```json
{
  "device_id": "ACP-SP2617XXXXX1",
  "agent_id": "agent_001",
  "call_uuid": "CALL-XXXX",
  "status": "reserved",
  "reserved_at": "2026-06-09T10:00:00.000Z",
  "expires_at": "2026-06-09T10:00:10.000Z"
}
```

## 超时建议

默认建议占用超时时间为 10 秒。这个值不是协议强制要求，接入方可以根据业务调整。

常见做法：

- `reserved` 阶段较短，例如 10 秒。超过后仍未发布 CALL，自动释放。
- `calling` 阶段按呼叫等待策略设置，例如 60 秒。
- `in_call` 阶段由通话状态、RTC 事件和服务端兜底任务释放。

## 释放时机

以下情况都必须释放设备占用：

- 座席取消呼叫。
- CALL 发布失败。
- 呼叫超时。
- 收到 `HANGUP`。
- 收到 `ERROR`。
- RTC 加入失败并终止呼叫。
- Web 页面关闭或后端检测到会话失效。

设备占用由接入方后端实现。当前文档不假设设备管理平台提供占用/释放 Open API。
